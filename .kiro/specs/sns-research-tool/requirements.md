# Requirements: sns-research-tool

> SNSリサーチ分析ツールの要件定義書（EARS準拠・C.U.T.E.採点対応）

## 1. 目的

マーケティング担当者およびSNS運用担当者が、複数SNSプラットフォームのアカウント分析・競合分析・キーワード分析を一元的に実行し、データに基づいた意思決定を可能にすること。

## 2. 概要（Executive Summary）

複数SNSプラットフォーム（X, Instagram, TikTok）のアカウント分析・競合分析・キーワード分析を一元的に実行するWebアプリケーションである。マーケティング担当者およびSNS運用担当者がデータに基づいた意思決定を行うために使用する。

## 3. 背景 & Context

- 現状、SNS分析は各プラットフォームの管理画面を個別に確認する必要があり、作業に1日あたり2時間以上を要している。
- 競合分析は手動でスプレッドシートに記録しており、データの鮮度と正確性に課題がある。
- レポート作成に毎週4時間以上を費やしており、自動化の余地がある。

## 4. スコープ

### 4.1 In Scope
- X (Twitter) API v2 を利用したデータ取得
- Instagram Graph API を利用したデータ取得
- TikTok Research API を利用したデータ取得
- アカウント分析（フォロワー推移、エンゲージメント率、投稿頻度）
- 競合アカウント比較分析
- キーワード・ハッシュタグ分析
- センチメント分析（ポジティブ/ネガティブ/ニュートラル分類）
- ダッシュボード表示
- PDF/Excelレポート出力
- マルチユーザー認証・チーム機能

### 4.2 Out of Scope（重要）
- Facebook / LinkedIn / YouTube の分析
- 投稿予約・自動投稿機能
- DM自動返信機能
- 広告運用・広告分析機能
- インフルエンサーマッチング機能
- AIによる投稿文生成機能

## 5. 用語集 / Glossary

| 用語 | 定義 |
|------|------|
| エンゲージメント率 | (いいね数 + コメント数 + シェア数) / フォロワー数 × 100 (%) |
| インプレッション | 投稿がユーザーの画面に表示された回数 |
| リーチ | 投稿を閲覧したユニークユーザー数 |
| センチメント | テキストの感情傾向（Positive: 0.5以上, Negative: -0.5以下, Neutral: -0.5〜0.5） |
| 分析対象アカウント | ユーザーが登録した自社または競合のSNSアカウント |
| データ取得ジョブ | APIからデータを取得するバッチ処理 |
| レポート | 分析結果をPDFまたはExcel形式で出力したドキュメント |

## 6. ステークホルダー & 役割

| Role | 権限/責務 |
|------|----------|
| システム管理者 | ユーザー管理、APIキー設定、システム設定変更 |
| チーム管理者 | チームメンバー管理、分析対象アカウント登録、レポート設定 |
| 一般ユーザー | ダッシュボード閲覧、レポート出力、キーワード分析実行 |
| ゲストユーザー | 共有されたレポートの閲覧のみ |

## 7. 前提/仮定（Assumptions）

- 【仮定】各SNSプラットフォームのAPI利用規約に準拠し、必要なAPI認証情報を取得済みであること
- 【仮定】X API v2 Basic プラン（月額100ドル相当）のレート制限内で運用すること
- 【仮定】Instagram Graph API はビジネスアカウントまたはクリエイターアカウントのみ対象とすること
- 【仮定】TikTok Research API の利用承認を取得済みであること
- 【仮定】初期ユーザー数は100名以下、分析対象アカウント数は500件以下を想定
- 【仮定】日本語および英語のセンチメント分析に対応すること

## 8. 制約（Constraints）

- X API v2 Basic: 10,000 tweets/月、50 requests/15min
- Instagram Graph API: 200 calls/hour/user
- TikTok Research API: 1,000 requests/day
- 実行環境: AWS (ap-northeast-1) または GCP (asia-northeast1)
- 個人情報保護法に準拠し、取得データは日本国内のリージョンに保存すること
- APIキーは環境変数または AWS Secrets Manager で管理すること

## 9. 成功条件（Success Metrics）

| Metric | Target | Measurement Method |
|--------|--------|-------------------|
| 分析作業時間削減率 | 70%以上 | 導入前後の作業時間比較 |
| レポート作成時間 | 10分以内 | レポート生成開始から完了までの時間計測 |
| データ取得成功率 | 99%以上 | 成功ジョブ数 / 全ジョブ数 × 100 |
| ユーザー満足度 | NPS 40以上 | 四半期アンケート |

## 10. 機能要件（Functional Requirements）

### REQ-001: ユーザー認証

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがログインフォームにメールアドレスとパスワードを入力して送信ボタンをクリックしたとき、システムは入力値を検証し、認証成功の場合は3秒以内にダッシュボード画面へ遷移しなければならない。
- 根拠/目的: 不正アクセス防止とユーザー識別のため
- 受入テスト(GWT):
  - AT-001: Given 登録済みユーザー(email: test@example.com, password: ValidPass123!)が存在する When ログインフォームに正しい認証情報を入力して送信する Then 3秒以内にダッシュボード画面が表示され、HTTPステータス200が返却される
  - AT-002: Given 登録済みユーザーが存在する When 誤ったパスワードを5回連続で入力する Then アカウントが30分間ロックされ、エラーメッセージ「アカウントがロックされました。30分後に再試行してください。」が表示される
- 例外・エラー:
  - EH-001: If 認証情報が一致しない場合、システムはエラーメッセージ「メールアドレスまたはパスワードが正しくありません」を表示し、入力フォームを再表示しなければならない
  - EH-002: If 認証サーバーが5秒以内に応答しない場合、システムはエラーメッセージ「認証サーバーに接続できません。しばらく待ってから再試行してください。」を表示しなければならない
- 補足:
  - 関連: REQ-002, REQ-003
  - 備考: OAuth2.0 (Google, Microsoft) によるSSO対応は Phase 2 で検討

### REQ-002: 分析対象アカウント登録

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): チーム管理者がアカウント登録フォームにSNSプラットフォーム種別とアカウントIDを入力して登録ボタンをクリックしたとき、システムは該当APIを呼び出してアカウントの存在を検証し、存在する場合は5秒以内にデータベースに登録しなければならない。
- 根拠/目的: 分析対象を明確に定義し、データ取得の対象を管理するため
- 受入テスト(GWT):
  - AT-003: Given チーム管理者としてログイン済みである When プラットフォーム「X」、アカウントID「@twitter」を入力して登録ボタンをクリックする Then 5秒以内に「アカウントを登録しました」メッセージが表示され、アカウント一覧に該当アカウントが表示される
  - AT-004: Given チーム管理者としてログイン済みである When 既に登録済みのアカウントIDを入力して登録ボタンをクリックする Then エラーメッセージ「このアカウントは既に登録されています」が表示される
- 例外・エラー:
  - EH-003: If 指定されたアカウントIDがSNSプラットフォーム上に存在しない場合、システムはエラーメッセージ「指定されたアカウントが見つかりません。アカウントIDを確認してください。」を表示しなければならない
  - EH-004: If APIレート制限に到達した場合、システムはエラーメッセージ「API制限に達しました。次回実行可能時刻に再試行してください。」を表示しなければならない
- 補足:
  - 関連: REQ-003, REQ-004
  - 備考: 1チームあたりの登録上限は50アカウント

### REQ-003: データ取得ジョブ実行

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): 毎日午前6時（JST）になったとき、システムは登録済みの全アカウントに対してデータ取得ジョブを実行し、各アカウントの過去24時間分の投稿データ、フォロワー数、エンゲージメントデータを取得してデータベースに保存しなければならない。
- 根拠/目的: 日次でデータを蓄積し、時系列分析を可能にするため
- 受入テスト(GWT):
  - AT-005: Given 分析対象アカウントが10件登録されている When 午前6時になる Then 60分以内に全10件のアカウントのデータ取得が完了し、データベースに保存される
  - AT-006: Given データ取得ジョブが実行中である When 同一アカウントに対する重複実行リクエストが発生する Then 重複実行は拒否され、ログに「ジョブ重複スキップ: アカウントID」が記録される
- 例外・エラー:
  - EH-005: If 特定アカウントのデータ取得に失敗した場合、システムは該当アカウントをスキップして次のアカウントの処理を継続し、失敗したアカウントのリストをログに記録しなければならない
  - EH-006: If APIレート制限に到達した場合、システムは残りのアカウントの処理を次回実行時刻（15分後）まで延期しなければならない
- 補足:
  - 関連: REQ-002, REQ-004
  - 備考: 手動実行機能はREQ-004で定義

### REQ-004: 手動データ取得

- 種別: EARS-イベント駆動
- 優先度: SHOULD
- 要件文(EARS): ユーザーがダッシュボード上の「今すぐ更新」ボタンをクリックしたとき、システムは選択されたアカウントの最新データを取得し、30秒以内にダッシュボードの表示を更新しなければならない。
- 根拠/目的: 緊急時やリアルタイム確認が必要な場合に対応するため
- 受入テスト(GWT):
  - AT-007: Given ダッシュボードにアカウント「@example」が表示されている When 「今すぐ更新」ボタンをクリックする Then 30秒以内にデータが更新され、最終更新時刻が現在時刻に変更される
- 例外・エラー:
  - EH-007: If 直近15分以内に同一アカウントの手動更新が実行されていた場合、システムはエラーメッセージ「前回の更新から15分以上経過してから再実行してください。」を表示しなければならない
- 補足:
  - 関連: REQ-003
  - 備考: 1ユーザーあたり1日20回の手動更新制限あり

### REQ-005: フォロワー推移グラフ表示

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがダッシュボード上でアカウントを選択したとき、システムは過去90日間のフォロワー数推移を折れ線グラフで表示し、日次の増減数を数値で表示しなければならない。
- 根拠/目的: フォロワー成長の傾向を視覚的に把握するため
- 受入テスト(GWT):
  - AT-008: Given アカウント「@example」に90日分のフォロワーデータが存在する When ダッシュボードで該当アカウントを選択する Then 2秒以内に90日間の折れ線グラフが表示され、各日の増減数がツールチップで確認できる
  - AT-009: Given アカウント「@newaccount」に30日分のデータしか存在しない When ダッシュボードで該当アカウントを選択する Then 30日分のグラフが表示され、データがない期間はグレーアウトされる
- 例外・エラー:
  - EH-008: If フォロワーデータが1件も存在しない場合、システムは「データがありません。データ取得完了後に再度アクセスしてください。」メッセージを表示しなければならない
- 補足:
  - 関連: REQ-006
  - 備考: 期間選択（7日/30日/90日/1年）のフィルタ機能を提供

### REQ-006: エンゲージメント率計算・表示

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムは各投稿のエンゲージメント率を (いいね数 + コメント数 + シェア数) / 投稿時点のフォロワー数 × 100 の計算式で算出し、小数点第2位まで表示しなければならない。
- 根拠/目的: 投稿パフォーマンスを標準化された指標で比較するため
- 受入テスト(GWT):
  - AT-010: Given 投稿Aのいいね数=100, コメント数=20, シェア数=10, 投稿時フォロワー数=10000 When エンゲージメント率を計算する Then エンゲージメント率=1.30%と表示される
- 例外・エラー:
  - EH-009: If フォロワー数が0の場合、システムはエンゲージメント率を「N/A」と表示しなければならない
- 補足:
  - 関連: REQ-005, REQ-007
  - 備考: プラットフォームごとのエンゲージメント指標の差異を用語集で定義

### REQ-007: 競合比較レポート生成

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがレポート生成画面で自社アカウント1件と競合アカウント1〜5件を選択してレポート生成ボタンをクリックしたとき、システムは60秒以内にPDFまたはExcel形式の比較レポートを生成し、ダウンロードリンクを表示しなければならない。
- 根拠/目的: 競合との比較分析を資料化し、社内共有・意思決定に活用するため
- 受入テスト(GWT):
  - AT-011: Given 自社アカウント1件と競合アカウント3件が選択されている When PDF形式でレポート生成ボタンをクリックする Then 60秒以内にPDFファイルのダウンロードリンクが表示され、ファイルサイズが10MB以下である
  - AT-012: Given レポート生成が完了した When ダウンロードリンクをクリックする Then PDFファイルがダウンロードされ、フォロワー推移比較・エンゲージメント率比較・投稿頻度比較の3セクションが含まれている
- 例外・エラー:
  - EH-010: If レポート生成に60秒以上かかる場合、システムはバックグラウンド処理に切り替え、完了時にメール通知を送信しなければならない
- 補足:
  - 関連: REQ-005, REQ-006
  - 備考: レポートテンプレートのカスタマイズ機能は Phase 2 で検討

### REQ-008: キーワード・ハッシュタグ分析

- 種別: EARS-イベント駆動
- 優先度: SHOULD
- 要件文(EARS): ユーザーがキーワード分析画面でキーワードまたはハッシュタグと分析期間を指定して分析開始ボタンをクリックしたとき、システムは該当キーワードを含む投稿を検索し、投稿数推移・センチメント分布・関連キーワードを120秒以内に表示しなければならない。
- 根拠/目的: 特定トピックに関する市場動向・ユーザー反応を把握するため
- 受入テスト(GWT):
  - AT-013: Given キーワード「#AI」、期間「7日」が指定されている When 分析開始ボタンをクリックする Then 120秒以内に投稿数推移グラフ、センチメント円グラフ（Positive/Negative/Neutral）、関連キーワードリスト（上位20件）が表示される
- 例外・エラー:
  - EH-011: If 検索結果が0件の場合、システムは「該当するデータが見つかりませんでした。キーワードまたは期間を変更してください。」メッセージを表示しなければならない
  - EH-012: If APIレート制限に到達した場合、システムは「API制限に達しました。次回実行可能時刻に再試行してください。」メッセージを表示し、部分的な結果がある場合はそれを表示しなければならない
- 補足:
  - 関連: REQ-009
  - 備考: X API の検索機能を使用、過去7日間の制限あり

### REQ-009: センチメント分析

- 種別: EARS-普遍
- 優先度: SHOULD
- 要件文(EARS): システムは取得した投稿テキストに対して自然言語処理によるセンチメント分析を実行し、各投稿を Positive（スコア0.5以上）、Negative（スコア-0.5以下）、Neutral（スコア-0.5超0.5未満）の3カテゴリに分類しなければならない。
- 根拠/目的: ユーザー反応の感情傾向を定量的に把握するため
- 受入テスト(GWT):
  - AT-014: Given 投稿テキスト「この製品は最高です！大満足！」が存在する When センチメント分析を実行する Then 該当投稿は Positive に分類され、スコアが0.5以上である
  - AT-015: Given 投稿テキスト「ひどいサービスだ。二度と使わない。」が存在する When センチメント分析を実行する Then 該当投稿は Negative に分類され、スコアが-0.5以下である
- 例外・エラー:
  - EH-013: If 投稿テキストが空または画像のみの場合、システムはセンチメントを「Unknown」として記録しなければならない
- 補足:
  - 関連: REQ-008
  - 備考: 日本語・英語に対応、他言語は Phase 2 で検討

### REQ-010: チーム機能

- 種別: EARS-イベント駆動
- 優先度: SHOULD
- 要件文(EARS): チーム管理者がチーム設定画面でメールアドレスを入力して招待ボタンをクリックしたとき、システムは招待メールを60秒以内に送信し、招待されたユーザーがリンクをクリックして登録を完了すると、該当チームのメンバーとして追加しなければならない。
- 根拠/目的: 複数メンバーでの共同作業を可能にするため
- 受入テスト(GWT):
  - AT-016: Given チーム管理者としてログイン済みである When メールアドレス「new@example.com」を入力して招待ボタンをクリックする Then 60秒以内に招待メールが送信され、ステータスが「招待中」と表示される
  - AT-017: Given 招待メールを受信している When メール内のリンクをクリックしてパスワードを設定する Then チームメンバーとして登録され、ダッシュボードにアクセスできる
- 例外・エラー:
  - EH-014: If 招待メールの送信に失敗した場合、システムはエラーメッセージ「メール送信に失敗しました。メールアドレスを確認して再試行してください。」を表示しなければならない
  - EH-015: If 招待リンクの有効期限（72時間）が切れている場合、システムは「招待リンクの有効期限が切れています。管理者に再招待を依頼してください。」メッセージを表示しなければならない
- 補足:
  - 関連: REQ-001
  - 備考: 1チームあたり20名まで

## 11. 非機能要件（Non-Functional Requirements）

### REQ-901: レスポンス時間

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはダッシュボード画面の初期表示を3秒以内に完了しなければならない。
- 根拠/目的: ユーザー体験の維持と離脱防止のため
- 受入テスト(GWT):
  - AT-018: Given ユーザーがログイン済みである When ダッシュボードURLにアクセスする Then 3秒以内に画面が表示され、主要データ（フォロワー数、直近投稿）が表示される
- 例外・エラー:
  - EH-016: If 3秒以内にデータ取得が完了しない場合、システムはスケルトンUIを表示し、データ取得完了後に実データで置換しなければならない
- 補足:
  - 備考: CDN使用、キャッシュ戦略を設計書で定義

### REQ-902: 可用性

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムは月間稼働率99.5%以上を維持しなければならない。
- 根拠/目的: ビジネスクリティカルな分析業務の継続性を保証するため
- 受入テスト(GWT):
  - AT-019: Given 30日間の運用期間がある When 可用性を計測する Then 稼働時間 / 総時間 × 100 が 99.5% 以上である
- 例外・エラー:
  - EH-017: If システム障害が発生した場合、システムは15分以内にアラートを発報し、運用チームに通知しなければならない
- 補足:
  - 備考: マルチAZ構成を前提、計画メンテナンスは月4時間まで

### REQ-903: データ保持期間

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムは取得したSNSデータを2年間保持し、2年経過後は自動的に削除しなければならない。
- 根拠/目的: ストレージコスト削減とデータ保護法令遵守のため
- 受入テスト(GWT):
  - AT-020: Given 2年以上前のデータが存在する When 日次バッチ処理が実行される Then 2年以上前のデータが削除され、削除件数がログに記録される
- 例外・エラー:
  - EH-018: If データ削除に失敗した場合、システムはエラーをログに記録し、翌日の実行時に再試行しなければならない
- 補足:
  - 備考: 削除前のバックアップは行わない

### REQ-904: 同時接続ユーザー数

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムは同時接続ユーザー50名まで、レスポンス時間3秒以内を維持しなければならない。
- 根拠/目的: 想定ユーザー数に対応したキャパシティを確保するため
- 受入テスト(GWT):
  - AT-021: Given 負荷テスト環境が準備されている When 50名の同時アクセスをシミュレートする Then 全ユーザーのダッシュボード表示が3秒以内に完了する
- 例外・エラー:
  - EH-019: If 同時接続数が50名を超えた場合、システムは新規接続に対して「現在アクセスが集中しています。しばらく待ってから再度アクセスしてください。」メッセージを表示しなければならない
- 補足:
  - 備考: オートスケーリング設定は設計書で定義

## 12. セキュリティ/プライバシー要件

### REQ-SEC-001: 認証トークン管理

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはJWTアクセストークンの有効期限を15分、リフレッシュトークンの有効期限を7日間に設定しなければならない。
- 根拠/目的: セッションハイジャックリスクの低減と利便性のバランス
- 受入テスト(GWT):
  - AT-022: Given ユーザーがログインした When 15分経過後にAPIリクエストを送信する Then アクセストークン期限切れエラー（401）が返却され、リフレッシュトークンで新しいアクセストークンを取得できる
- 例外・エラー:
  - EH-020: If リフレッシュトークンも期限切れの場合、システムはユーザーをログイン画面にリダイレクトしなければならない

### REQ-SEC-002: APIキー暗号化

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはSNSプラットフォームのAPIキーをAES-256で暗号化してデータベースに保存しなければならない。
- 根拠/目的: APIキー漏洩時の被害を最小化するため
- 受入テスト(GWT):
  - AT-023: Given APIキーがデータベースに保存されている When データベースを直接参照する Then APIキーは暗号化された状態で保存されており、平文では読み取れない
- 例外・エラー:
  - EH-021: If 暗号化処理に失敗した場合、システムはAPIキーを保存せず、エラーメッセージ「APIキーの保存に失敗しました。」を表示しなければならない

### REQ-SEC-003: 監査ログ

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはログイン、ログアウト、アカウント登録、レポート生成、設定変更の操作を監査ログとして記録し、90日間保持しなければならない。
- 根拠/目的: セキュリティインシデント発生時の調査と証跡確保のため
- 受入テスト(GWT):
  - AT-024: Given ユーザーがログインした When 監査ログを確認する Then ログインイベント（タイムスタンプ、ユーザーID、IPアドレス、User-Agent）が記録されている
- 例外・エラー:
  - EH-022: If 監査ログの書き込みに失敗した場合、システムはエラーをアラートとして発報し、操作自体は継続しなければならない

## 13. ログ/監視/運用要件

### REQ-OPS-001: エラーアラート

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): システムでHTTP 5xxエラーが5分間に10回以上発生したとき、システムは運用チームにSlackアラートを送信しなければならない。
- 根拠/目的: 障害の早期検知と対応のため
- 受入テスト(GWT):
  - AT-025: Given 監視設定が有効である When 5分間に5xxエラーが10回発生する Then Slackの#alertsチャンネルにアラートメッセージが投稿される
- 例外・エラー:
  - EH-023: If Slack通知に失敗した場合、システムはメールでバックアップ通知を送信しなければならない

### REQ-OPS-002: ログレベル設定

- 種別: EARS-普遍
- 優先度: SHOULD
- 要件文(EARS): システムは環境変数LOG_LEVELによりログ出力レベル（DEBUG/INFO/WARN/ERROR）を制御可能でなければならない。
- 根拠/目的: 本番環境とデバッグ環境でのログ量の調整と、障害調査時の詳細ログ取得のため
- 受入テスト(GWT):
  - AT-026: Given LOG_LEVEL=ERROR が設定されている When INFO レベルのログ出力コードが実行される Then ログは出力されない
  - AT-027: Given LOG_LEVEL=DEBUG が設定されている When DEBUG レベルのログ出力コードが実行される Then ログが出力される
- 例外・エラー:
  - EH-024: If LOG_LEVELが未設定または不正な値の場合、システムはデフォルト値INFOを使用しなければならない
